# Risks And Open Questions

## Real Risks

### Specs Can Be Too Vague

If the spec does not say what matters, tests can pass while behavior is wrong.
The conformance gate only certifies the criteria it was given.

Mitigation:

- Require acceptance tests for important workflows.
- Add a supervisor that detects repeated ambiguity.
- Keep specs reviewed and concrete.

### Generated Internals Can Become Accidental APIs

If downstream code depends on generated file layout, helper names, or incidental
implementation details, regeneration becomes dangerous.

Mitigation:

- Only depend on spec-defined interfaces.
- Keep generated code behind stable public contracts.
- Avoid hand-editing generated output.

### Tight Coupling Breaks Slice Boundaries

The model works best when modules have clean boundaries. It gets harder when a
feature depends on subtle behavior across many generated files.

Mitigation:

- Keep functional units small.
- Use import modules for shared definitions.
- Use requires only for same-stack build chains.
- Make cross-stack boundaries explicit through schemas or APIs.

### The Renderer Can Drift

In retry loops, a renderer may weaken tests or change implementation in a way
that satisfies the immediate gate but violates intent.

Mitigation:

- Diff conformance tests between attempts.
- Stop loops when assertions get weaker.
- Treat generated tests as generated artifacts, not unquestionable truth.

### Test Scripts Become The Hidden Contract

If the script swallows failures, misses tests, or stages files incorrectly, the
renderer receives bad feedback.

Mitigation:

- Read scripts before rendering.
- Make healthcheck validate config and script presence.
- Run scripts manually when diagnosing failures.

### Cost Can Hide In Retry Loops

The public product bills successful functionalities, but local systems still pay
model/API cost for failed attempts. Long loops can waste time and money.

Mitigation:

- Use early stop thresholds.
- Log every attempt.
- Prefer focused render ranges.

## Strong Fits

This approach seems strongest for:

- external API integrations
- SDK/client generation
- schema-heavy backends
- command-line tools
- internal tools with clear workflows
- systems where behavior is stable but APIs or implementation details change

## Weak Fits

This approach is harder for:

- highly visual UI polish with subjective taste
- performance-critical systems needing hand tuning
- systems with many implicit cross-module dependencies
- legacy systems where generated code must preserve undocumented internals
- safety-critical software without strong external verification

## Open Questions For Our Prototype

1. Should we use Plain-compatible syntax or define a simpler `mint` spec format?
2. What first target stack should we support?
3. Should generated output live in the main repo or a sub-repo under
   `generated/<module>`?
4. Do we use Git for checkpoints, or a custom metadata plus patch store?
5. Which model/provider should handle rendering?
6. Should conformance tests be generated by the same model call as code or a
   separate call?
7. How strict should we be about generated code being read-only?
8. What is our minimum healthcheck before a render?
9. What retry thresholds should stop the loop?
10. What is the first real test case: API integration, CLI tool, or small web
    service?

## Decision Bias

Start narrower than Codeplain:

- one stack
- one module
- one test framework
- one generated folder
- one healthcheck command
- explicit acceptance tests

Once the loop is proven, add graph rendering and richer authoring skills.
