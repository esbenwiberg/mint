# Spec authoring guide

How to write `.mint.md` specs that render reliably, re-render cheaply, and
survive being regenerated by a different model on a different day. Everything
here was learned the hard way on `examples/timesheet` — a six-module graph
(store, rules, HTTP API, CLI, design system, web UI) that has been through a
full cross-cutting rewrite and several targeted tweaks.

## The one-sentence mental model

A spec is a contract, and the renderer is an adversary who will honor exactly
what you pinned and improvise everything you didn't. If a detail matters,
pin it in a bullet a test can check; if it doesn't, leave it out and accept
that it may change on every re-record.

## Know what each section costs

Mint decides what to re-render by hashing spec sections separately. Where you
put a sentence determines the blast radius of changing it later:

| You change | What re-renders |
|---|---|
| `## definitions`, `## implementation`, `## test` (shared sections) | the **whole module**, FR1 onward |
| one unit's `spec:`/`acceptance:` bullets | that unit **and every unit after it** |
| the last unit's bullets | just that unit |
| a required module's **public interface** (exported names, signatures, docstrings, literal constants) | that module, plus **every dependent module in full** |
| a required module's internals (bodies, private helpers) | that module only — dependents NOOP |

Consequences worth designing for:

- **Put a constraint in the narrowest section that owns it.** "The date field
  has `type="date"`" belongs in the add-form unit's bullets (a 2-unit
  re-record), not in the shared definitions (a whole-module re-record). We
  measured exactly this on `timesheet-web`: same sentence, 2 units vs 4.
- **Order units by stability.** Contracts that will not move (routes, error
  codes, exit codes) go in early units; churn-prone details (UI copy, layout)
  go late, where edits are cheapest. Changing FR1 re-renders everything after
  it, because each unit's render prompt embeds the code the earlier units
  produced.
- **Leaf modules should hoard interface stability.** A dependency cascades
  only through what dependents can see. Renaming a private helper is free;
  changing a public constant re-renders every dependent. When we removed the
  `person` field across the whole timesheet graph, the `rules` module — whose
  spec we deliberately left untouched — NOOPed through the entire cascade and
  its cassettes survived byte-identical in the diff. Mint watches this for
  you: a completed render reports when the public interface changed (with the
  added/removed names and the dependents that will re-render), and warns when
  a public constant carries a large literal — content like a stylesheet or
  template riding the cascade belongs behind a private literal + accessor.

## Write acceptance a test can grip

Every acceptance bullet must name an observable: a return value, an exit code,
a status code, a stored state, an output substring, a raised error with a
message fragment. `mint lint` rejects vague phrasing, but passing lint is the
floor, not the bar:

- **Name exact values.** `responds 422 and the body contains a ts-error
  element whose text contains "week"` — not "shows a helpful error".
- **Exercise boundaries explicitly.** If a spec bullet states a threshold
  ("beyond the 24-hour cap"), lint requires an acceptance bullet that tests
  the exact boundary. Phrase it mid-sentence: `posting 4.0 more hours responds
  303 — the exact 24-hour day boundary — and posting 4.5 instead responds 422`.
  A bare number followed by a period can slip past lint's boundary detector.
- **Watch angle-bracket-digit literals.** `<h1>` in a spec bullet parses as
  the relational threshold "1 >" and lint will demand a boundary test for it.
  Write "the heading" or `<h1` without the closing bracket.
- **Include the sad paths.** Unknown ids, invalid input, illegal transitions,
  empty results. Lint warns when a spec has no edge-case coverage at all, and
  a renderer given only happy-path acceptance will write only happy-path code.

## Pin contracts, not implementations

The spec pins the public surface; the renderer owns everything behind it —
and *will* legitimately reorganize it between recordings. On one re-record,
`timesheet-cli`'s `main()` moved from `cli.py` into `__init__.py`. Perfectly
legal: the spec pins the package name and `main(argv=None)`, not the file
layout. It broke our hand-written wrapper, which had hardcoded
`python -m timesheet_cli.cli`.

The rule for anything hand-written around generated modules (scripts, hooks,
CI): **glue may only depend on spec-pinned surface.** If glue needs a
guarantee the spec doesn't give, add the guarantee to the spec — don't couple
to whatever the current recording happens to look like.

## UI modules: the style lock

Aesthetics are underdetermined by prose — "a clean modern layout" produces a
different app every recording, and screenshot tests fail every legitimate
re-render by design. Split the problem:

1. **Skin lives in a dedicated module** (`ui-kit`): a design-token stylesheet
   kept in a **private literal** (`_TOKENS_CSS`) behind a **public accessor**
   (`tokens_css()`). Pin exact values (`--ts-accent: #4f46e5`) in acceptance
   bullets so every re-record reproduces them byte-for-byte. Privacy matters
   for the cascade: interface stubs drop private names and function bodies, so
   restyling re-renders only the kit — dependents NOOP and their cassettes
   survive. They don't need the values in their prompt anyway: their code
   calls the page shell, which embeds the stylesheet **at runtime**, so
   rendered pages pick up a restyle without regenerating a line of dependent
   code. (The first version exported the stylesheet as a public literal; every
   color tweak re-rendered every dependent in full, for output that couldn't
   differ. Don't repeat that.) What dependents *do* need pinned is the **class
   contract** — names and composition rules — which lives in their own spec
   bullets and the accessor's docstring.
2. **Structure is pinned as a DOM contract** in the consuming module: routes,
   `data-testid` hooks, escaped output, status codes — asserted by conformance
   tests. Pixels are never asserted.
3. **Forbid freelance styling mechanically.** Declare the lock in frontmatter
   and mint enforces it — no prose bullets or generated scans to rely on:

   ```yaml
   styleLock:
     kit: ui-kit
     classPrefix: ts-
   ```

   Mint scans the generated `src/` on every render attempt for `<style`
   elements, `style=` attributes, and class tokens outside the prefix, fails
   the attempt with the offending lines as retry feedback, and injects the
   constraint into every render prompt. The renderer literally cannot ship its
   own look. Acceptance bullets asserting the same things remain useful as
   documentation of the contract, but they are no longer the enforcement.
4. **Spec the composition rule.** CSS modifier classes only work stacked on
   their base class (`class="ts-button ts-button--danger"`). Our first
   recording used a variant class alone and got browser-default borders. If a
   class system has usage rules, they are contract — pin them.

## The iteration loop: a ratchet, not a sketchpad

Design and debugging are exploratory: you try ten variants to keep one. The
render loop is the opposite — every spec edit costs live model calls, orphans
cassettes, and re-rolls everything you didn't pin. Iterating *through* the
renderer means paying blast-radius prices for throwaway guesses. Don't. Split
the work into two loops:

**Inner loop — explore outside the renderer, zero renders.** The generated
tree is ordinary code; edit it directly and run the app. For UI work this is
especially cheap: because kit classes only reference `var(--ts-*)` tokens, you
can restyle the entire app live in browser devtools by overriding the `:root`
block, or hand-edit the generated stylesheet and reload. The offline
conformance tests still run against a hand-edited tree, so you always know
whether an experiment left the contract. `mint drift <module>` shows
everything you've changed since the last checkpoint.

**Outer loop — commit the settled result through the spec, once.** When the
exploration converges, translate the surviving decisions into spec bullets
(narrowest owning section — see the blast-radius table), check
`mint status <module>` to confirm the cost matches your intent, re-record,
verify an offline replay NOOPs, and prune.

The one hard rule that makes this safe: **hand edits are ephemeral.** The next
render resets the tree to spec; anything not captured in a bullet is gone by
design. Treat `mint drift` output as the backport checklist before you
re-record — and treat lingering drift as a smell, the same way you'd treat an
uncommitted working tree at the end of the day.

The mental model: the spec loop is a ratchet, not a sketchpad. Exploration is
high-frequency and low-commitment, so it happens outside the render loop;
rendering is low-frequency and high-commitment, so it happens once per settled
decision. If you find yourself paying for a render whose outcome you plan to
judge by eye and probably discard, you're sketching with the ratchet.

## The recording workflow

```bash
mint lint <module>                                # spec quality gate
mint doctor                                       # toolchain + replay state
MINT_LIVE=1 mint render <module> --range FR1:FR1  # record one unit at a time
mint render <module>                              # replay offline; must NOOP when repeated
mint prune                                        # list orphaned cassettes; --yes to delete
```

- **Record per unit, foreground, with pipefail.** One unit per invocation
  keeps failures cheap to retry and cassettes easy to review.
- **`mint status <module>` before recording** tells you exactly which units a
  spec edit invalidated — check that the blast radius matches your intent
  before paying for it.
- **Verify replay before committing**: wipe the generated dirs, `mint render`
  the graph offline, and confirm a second render NOOPs. Committed cassettes
  are the build; if replay doesn't reproduce, the recording is broken.
- **Prune after re-records.** Every re-record orphans the replaced cassettes;
  nothing cleans them up automatically. `mint prune` refuses while any module
  is stale (the live cassette set is only knowable from an up-to-date render),
  lists orphans dry-run by default, and deletes with `--yes`. Re-verify a
  wipe-and-replay after pruning.
- **Check `git status` before committing.** The renderer occasionally writes
  stray copies of module files outside the patch roots. Diff anything
  unexpected against the canonical generated output and delete it.

## Miscellany that bites

- Third-party deps are declared in the generated `pyproject.toml` /
  `package.json` — say so in the spec (`Declare fastapi and httpx in ...`),
  and remember FastAPI form posts also need `python-multipart`.
- Conformance tests drive code in-process (`TestClient`, direct imports) —
  never a server subprocess. Say that in the `## test` section or the renderer
  may spawn one.
- Fixtures in specs and tests are public forever (cassettes embed them).
  Use obviously fake data only.
