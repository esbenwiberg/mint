## medium: Cassette filenames collide across model names

id: fnd_sig-feat-cli-command-f51cbb7c31-_f2144a7d6b
category: data-loss
confidence: high
triage: confirmed-bug
status: open
feature: Python source src/mint_cli/renderer (feat_cli-command_f51cbb7c31)

evidence:
- src/mint_cli/renderer/cassettes.py:22-37 (cassette_id)
- src/mint_cli/renderer/cassettes.py:40-41 (cassette_path)
- src/mint_cli/renderer/cassettes.py:67-85 (RecordingClient.complete)
- src/mint_cli/renderer/cassettes.py:201-205 (ReplayClient._validate)

The cassette id includes prompt version, module, unit, phase, attempt, and prompt text, but not the configured model. Since the path is derived only from that id and RecordingClient writes the file unconditionally, recording the same request with a different model overwrites the previous cassette. Replay then validates the stored model and fails for the overwritten model, so one model's offline cassette corpus can silently destroy another's.

recommendation:
Include the model in the cassette key or namespace cassette paths by model, and add a migration/error path for old cassette ids if existing fixtures matter.

test analysis:
No linked tests exercise recording the same request under two model names or assert that model-specific cassettes can coexist.

suggested regression test:
Create two RecordingClient instances with different model values and the same cassette_dir/request/prompt; assert they write distinct cassette paths or that both ReplayClient(model=...) instances can replay successfully after both recordings.

minimum fix scope:
Change cassette_id/cassette_path inputs to account for model identity, update callers, and add coexistence coverage.

repro:
Record the same RenderRequest and prompt twice into the same cassette_dir with model='a' and model='b'. Both writes target the same v1/<hash>.json path, and replaying with model='a' after the second write raises the model mismatch error.

## medium: CLI subprocess test helper can hang the suite indefinitely

category: build-release
confidence: high
triage: risk
status: open

- tests/test_cli.py:16-33 (run_mint)

The main CLI test helper wraps subprocess.run without any timeout, so any regression that blocks in the CLI will block pytest/CI until the outer job times out. test analysis:
The tests exercise many CLI commands through this helper, but none assert that the helper bounds execution time or fails fast for a hung child process.

suggested regression test:
Add a small helper-level test that monkeypatches subprocess.run or invokes a controlled long-running command path and asserts run_mint passes a timeout argument.

minimum fix scope:
Update tests/test_cli.py::run_mint to pass timeout=... to subprocess.run.

## medium: Invalid config scalar values escape MintError handling

id: fnd_sig-feat-cli-command-832a326ac4-_d34bae2641
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Python source src/mint_cli (feat_cli-command_832a326ac4)

evidence:
- src/mint_cli/config.py:76-105 (load_config)
- src/mint_cli/config.py:107-108 (load_config)
- src/mint_cli/cli.py:271-275 (main)

load_config converts many user-controlled config values with int() and bool-like assumptions, but only catches KeyError. If mint.yaml contains an invalid scalar such as version: nope, maxFunctionalUnitsPerRender: many, or testQuality: false, load_config raises ValueError or AttributeError. cli.main only converts MintError into a user-facing error, so these bad config files produce an uncaught traceback instead of a clean FAIL/ERROR message.

recommendation:
Validate config field types and ranges explicitly, catch ValueError/TypeError/AttributeError during config construction, and re-raise MintError with the field name and expected type.

test analysis:
tests/test_config.py covers missing or unsafe output directories, but not malformed numeric/boolean config values or wrong-shaped sections.

suggested regression test:
Add tests for load_config with version: nope, limits.unitRetries: many, and testQuality: false, asserting MintError messages rather than raw Python exceptions.

minimum fix scope:
src/mint_cli/config.py load_config validation and error wrapping.

repro:
In mint.yaml, set version: nope and run mint doctor. The int(raw['version']) conversion raises ValueError, which is not wrapped in MintError and is not caught by cli.main.

## medium: Mutation probe treats a crashing conformance script as a killed mutant

id: fnd_sig-feat-test-suite-b323288fb4-d_4de0532cc9
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Python test suite src/mint_cli (feat_test-suite_b323288fb4)

evidence:
- src/mint_cli/test_quality.py:477-479 (_mutate_and_test)

A mutant is considered killed when either the unit script or conformance script exits non-zero. Because the conformance script is always executed and any non-zero exit counts as success for the probe, an unrelated broken or missing conformance runner can make every mutant look killed even when the unit tests still pass. That lets weak generated tests pass the mutation gate for the wrong reason.

recommendation:
Only count a mutant as killed by scripts that are configured and known to pass on the unmutated project, or run a baseline before mutation and treat baseline script failures as test-quality infrastructure failures rather than killed mutants.

test analysis:
This file contains the implementation but no included tests exercising mutation behavior when one project script fails independently of the mutation.

suggested regression test:
Add a test where the unit script returns 0, the conformance script returns non-zero before and after mutation, and assert the mutation probe reports an infrastructure failure instead of passed/killed.

minimum fix scope:
Update _mutate_and_test/_run_mutation_probe to distinguish baseline script failures from mutation-induced failures.

## medium: Build backend floor is too low for SPDX-style license metadata

id: fnd_sig-feat-library-c83dde8db7-e064_4ae84e7436
category: build-release
confidence: medium
triage: risk
status: open
feature: Python project mint-regenerative (feat_library_c83dde8db7)

evidence:
- pyproject.toml:2-3
- pyproject.toml:11

The project declares only setuptools>=68 as its build backend, but uses the string form of project.license. That form relies on newer SPDX-style project metadata support; older setuptools releases allowed by the declared floor may reject or mishandle it during isolated builds, making source/wheel builds depend on whatever backend version pip happens to resolve.

recommendation:
Either raise the build-system setuptools minimum to a version that supports string license expressions, or use the legacy PEP 621 table form compatible with the declared backend floor.

test analysis:
No packaging/build test is listed for this feature, and the README test command only runs pytest after an editable install rather than exercising isolated sdist/wheel metadata generation.

suggested regression test:
Add a packaging smoke test in CI that runs `python -m build` or equivalent isolated wheel/sdist build from a clean environment.

minimum fix scope:
pyproject.toml build-system/license metadata

## medium: Model renderer can return non-patch JSON as rendered output

id: fnd_sig-feat-cli-command-f51cbb7c31-_232f04adb4
category: api-contract
confidence: medium
triage: contract-mismatch
status: open
feature: Python source src/mint_cli/renderer (feat_cli-command_f51cbb7c31)

evidence:
- src/mint_cli/renderer/model.py:19-24 (SYSTEM_PROMPT)
- src/mint_cli/renderer/model.py:66-106 (ModelRenderer.render)
- src/mint_cli/renderer/model.py:171-177 (extract_json)
- src/mint_cli/renderer/patch.py:14-33 (validate_patch)

The renderer prompt requires a JSON object with a non-empty files list, and validate_patch implements that contract, but ModelRenderer only parses JSON and returns it as a rendered RenderOutcome. json.loads can return a list, string, or object missing files, so malformed model output can be classified as rendered and escape the ModelOutputError path with prompt/response/cassette context.

recommendation:
Validate the parsed object inside ModelRenderer.render before constructing RenderOutcome, and wrap validate_patch failures in ModelOutputError so callers get the same audit context as parse failures.

test analysis:
No linked tests cover model responses that are valid JSON but violate the patch schema.

suggested regression test:
Add a ModelRenderer test where the scripted response is '[]' or {'summary':'x','files':[]} and assert ModelOutputError includes the prompt, response, and cassette_id.

minimum fix scope:
Call validate_patch after extract_json in ModelRenderer.render, or make extract_json enforce object type and separately validate the patch schema before returning.

repro:
Use ScriptedModelClient({'default': '[]'}) with ModelRenderer.render(...). extract_json returns a list, and render returns RenderOutcome(patch=[], classification='rendered') instead of raising ModelOutputError.

## medium: Spec dependency names can escape project-scoped paths

id: fnd_sig-feat-cli-command-832a326ac4-_0813217f94
category: security
confidence: medium
triage: risk
status: open
feature: Python source src/mint_cli (feat_cli-command_832a326ac4)

evidence:
- src/mint_cli/specs.py:123-124 (parse_spec_file)
- src/mint_cli/workflow.py:204-211 (ModuleContext)
- src/mint_cli/workflow.py:289-299 (_spec_loader/resolve_required_order)
- src/mint_cli/workflow.py:307-323 (compute_context_hashes)

imports and requires are accepted as arbitrary strings, then used directly to build spec paths and generated source paths. A spec can name a dependency such as '../external' or '../specs/taskstore', causing Mint to read a spec outside the intended specs/<module>.mint.md namespace and to compute or pass generated paths outside generated/<module>. Because required modules also feed render ordering and runtime paths, this breaks the project directory boundary and can lead to reads or later writes in unintended locations.

recommendation:
Validate every imports/requires item with the same module slug rules used by new_module, reject path separators, '.', and '..', and have _spec_loader verify that the parsed spec.module exactly equals the requested module name.

test analysis:
The included specs tests cover self-require rejection and normal imports/requires parsing, but do not include invalid dependency names or traversal-like module references.

suggested regression test:
Add tests that parse or healthcheck specs with imports/requires values like '../base', 'base/child', '.', and '..', asserting MintError before any filesystem path is derived outside specs/.

minimum fix scope:
src/mint_cli/specs.py validation plus a defensive equality check in src/mint_cli/workflow.py _spec_loader.

repro:
Create a spec whose frontmatter contains requires: [../external] with ../external.mint.md present, or requires: [../specs/taskstore] in a project that has specs/taskstore.mint.md. Healthcheck/render dependency resolution will accept the reference string and derive paths from it instead of rejecting it as an invalid module slug.
